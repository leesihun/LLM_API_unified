# Slide Structure

- Title slide: `#` heading, plus a one-line subtitle when the topic warrants
  it. Optional `<!-- _class: lead -->` for centered styling.
- Body slides: `##` heading. One main idea per slide. 3–6 bullets, or a
  table, or a single visual — not all three.
- Closing slide: a takeaway, decision, or call-to-action. Skip only if the
  user explicitly asked for no conclusion.
- Separate slides with `---` on its own line.
- Do not exceed `{max_slides}` slides total, including title and closing.

# Narrative Arc

For decks longer than 4 slides, build a clear flow:

1. **Title** — what the deck is about.
2. **Context / problem** — why the audience should care.
3. **Body** — the core content, ordered by argument or chronology.
4. **Evidence** — data, examples, code, or visuals that support the body.
5. **Takeaway / next step** — the one thing the audience should leave with.

Pace the deck so no single section dominates unless the user asked for that.

# Slide Content Rules

- Bullets are short, parallel in form, and ordered by importance. Avoid full
  sentences when a noun phrase carries the meaning.
- Do not crowd a slide. If you have more than 6 bullets, split into two
  slides with continuation titles ("Topic (1/2)", "Topic (2/2)") or rework
  into a table.
- Use professional language and correct grammar.
- Preserve user-supplied technical terms, numbers, names, and quoted labels
  exactly.
- For code, fence with the correct language tag. Keep code blocks under ~12
  lines per slide; trim or split if longer.

# Marp Features You May Use

- Markdown headings, lists, bold, italic, tables, fenced code blocks.
- Images: `![](url)` for inline, `![bg](url)` for full-bleed, `![bg right]`
  for split layout. Always include alt text inside the brackets when the
  image carries meaning.
- Per-slide directives: `<!-- _class: lead -->`, `<!-- _backgroundColor:
  #123456 -->`, `<!-- _color: white -->`, `<!-- _paginate: false -->` for the
  title slide.
- Math: `$inline$` and `$$block$$`.
- Speaker notes: lines beginning with `<!--` on their own paragraph between
  slides; use sparingly and only when the user asked for them.

# Design Guidance

- Clarity over decoration. A clear bullet list beats a styled stub.
- Use concrete slide titles that communicate the slide's point ("Latency
  dropped 40% after batching"), not labels ("Results").
- Keep a consistent slide rhythm: heading, supporting points, optional
  visual.
- Avoid visual clutter, gratuitous emoji, and inline color CSS that fights
  the theme.
